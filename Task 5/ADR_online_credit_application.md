#### Цифровые каналы банка

Сайт и интернет-банк получают кредитный сценарий через уже реализованный в ADR_MVP гейтвей цифровых каналов. Каналы не обращаются напрямую к АБС и не берут на себя логику жизненного цикла заявки.

Доработки существующих компонентов:

- **Сайт банка** - показывает публичные кредитные продукты и принимает заявку нового клиента по Ф. И. О. и номеру телефона. Если клиент вводит паспортные данные, заявка может быть отправлена на предварительное решение.
- **Интернет-банк** - показывает доступные кредитные продукты, предодобренные предложения, счет зачисления средств и форму подачи заявки для действующего клиента.
- **Гейтвей цифровых каналов** - расширяется кредитными эндпоинтами. Он маршрутизирует запросы в сервис заявок, получает условия кредитных продуктов из Кредитного конвейера и возвращает клиенту статус заявки.

Новые инфраструктурные компоненты в цифровых каналах не добавляются. Используется существующая точка входа из ADR_MVP.

Основные интеграции:

- сайт и интернет-банк обращаются к гейтвею по HTTPS;
- гейтвей передает заявки в сервис заявок;
- гейтвей получает актуальные условия кредитных продуктов из Кредитного конвейера;
- гейтвей передает в интернет-банк данные о предодобренных предложениях, полученные из кредитного контура;
- интернет-банк не подключается напрямую к АБС, Кредитному конвейеру, БКИ и скорингу.

#### Расширение системы заявок

Система заявок уже реализована в ADR_MVP как единый владелец жизненного цикла заявки для цифровых каналов. В рамках этого ADR она расширяется типом кредитной заявки и передает заявку в АБС, чтобы менеджер фронт-офиса мог продолжить работу с ней в привычном контуре отделения.

Доработки существующих компонентов:

- **Сервис заявок** - добавляет кредитный тип заявки, хранит единый идентификатор, ведет статусы, передает заявку в АБС, инициирует процесс в Кредитном конвейере и запускает СМС-подтверждение для интернет-банка.
- **БД системы заявок** - расширяется полями для кредитной заявки: идентификаторы заявки в АБС и Кредитном конвейере, параметры кредитного продукта, признаки предодобрения, идемпотентные ключи операций и технические данные для восстановления обменов.

Новая отдельная система кредитных заявок не вводится. Кредитный сценарий использует уже созданную систему заявок из ADR_MVP.

Основные интеграции:

- принимает заявки от гейтвея цифровых каналов;
- передает заявку в АБС через существующую или доработанную внутреннюю интеграцию;
- инициирует процесс в Кредитном конвейере после создания заявки;
- получает статусы из Кредитного конвейера и делает их доступными сайту, интернет-банку и отделению;
- инициирует СМС-коды и СМС-уведомления через систему оповещений;
- принимает результаты предодобрений из изолированного контура скоринга и передает их в интернет-банк через гейтвей.

#### Доработки АБС

АБС сохраняется как учетная система и рабочий контур менеджера фронт-офиса в отделении. Уже существующий после ADR_MVP контролируемый интеграционный подход к АБС расширяется кредитными заявками: система заявок принимает заявку из цифровых каналов и передает ее в АБС, чтобы менеджер мог продолжить оформление там же.

Доработки существующих компонентов:

- **Прием онлайн-заявки** - АБС должна получать кредитную заявку из системы заявок с единым идентификатором и минимальным набором данных для продолжения оформления в отделении.
- **Поиск существующей заявки** - фронт-офис должен находить заявку в АБС по телефону, паспортным данным или идентификатору заявки.
- **Связь статусов** - изменения по заявке должны синхронизироваться между АБС, Кредитным конвейером и системой заявок.

Основные интеграции:

- сервис заявок передает онлайн-заявку в АБС;
- менеджер фронт-офиса продолжает заявку в АБС после идентификации клиента;
- если онлайн-заявки нет, менеджер может создать новую заявку в АБС по текущему процессу;
- существующая суточная передача из АБС в Кредитный конвейер сохраняется, но Кредитный конвейер должен исключать повторную обработку заявок, которые уже пришли онлайн быстрым путем.

#### Доработки Кредитного конвейера

Кредитный конвейер остается системой обработки кредитных заявок, но получает быстрый путь запуска процесса для онлайн-заявок. Это нужно, потому что текущая передача заявок из АБС раз в сутки не позволяет обработать предодобренные заявки в течение рабочего дня.

Доработки существующих компонентов:

- **BPM-процесс онлайн-заявки** - отдельный старт процесса из системы кредитных заявок через API или message start event.
- **Сервис обработчиков** - Java-обработчики, которые забирают детали онлайн-заявки, создают запись в штатной модели данных конвейера и фильтруют дубли из суточной загрузки АБС.
- **Рабочее место кредитного менеджера** - менеджер кредитного бэк-офиса продолжает работать в штатном интерфейсе Кредитного конвейера.

Основные интеграции:

- гейтвей цифровых каналов получает из Кредитного конвейера актуальные условия кредитных продуктов;
- сервис заявок инициирует запуск процесса онлайн-заявки;
- Кредитный конвейер запрашивает скоринг по заявке;
- Кредитный конвейер возвращает изменения статуса в систему заявок;
- суточный batch из АБС остается совместимым механизмом для старого процесса, но не является основным путем для онлайн-заявок.

#### Доработки системы кредитного скоринга

Основной скоринг не должен получать дополнительную массовую нагрузку от предрасчетов. Поэтому для предодобренных предложений выделяется изолированный контур, а основной сервис скоринга используется для расчета по конкретной заявке.

Доработки существующих компонентов:

- **Основной сервис кредитного скоринга** - рассчитывает решение по заявке по запросу Кредитного конвейера.
- **Сервис скоринга предодобрений** - изолированный контур для расчета предодобренных предложений действующим клиентам.
- **БД скоринга предодобрений** - хранит результаты предодобрений отдельно от основной БД скоринга.

Основные интеграции:

- Кредитный конвейер вызывает основной скоринг по конкретной заявке;
- основной скоринг получает данные БКИ для новых клиентов, если они ввели паспортные данные;
- сервис скоринга предодобрений получает данные о счетах и агрегированные признаки клиента из системы счетов;
- сервис скоринга предодобрений получает внешние данные из БКИ в контролируемом режиме;
- результаты предодобрений передаются в систему заявок и далее в интернет-банк через гейтвей.

#### Система оповещений

Система оповещений уже реализована в ADR_MVP для СМС-кодов, поэтому она переиспользуется для подтверждения подачи кредитной заявки в интернет-банке и для уведомления клиента об изменениях статуса.

Доработки существующих компонентов:

- **Сервис оповещений** - добавляет шаблоны и события для кредитных заявок, использует существующую логику СМС-кодов, срока жизни, повторной отправки и защиты от повторного использования.
- **БД сервиса оповещений** - дополняется шаблонами и журналами отправок по кредитным заявкам.

Основные интеграции:

- сервис заявок инициирует СМС-код при подаче заявки в интернет-банке;
- сервис заявок инициирует СМС-уведомление при изменении статуса заявки;
- сервис оповещений отправляет сообщения через существующий СМС-шлюз и телеком-оператора;
- результат отправки фиксируется для расследования ошибок и повторной доставки.

#### Интеграции с внешними и смежными системами

- **Бюро кредитных историй** используется как внешний источник данных для новых клиентов с сайта и для контролируемого расчета предодобренных предложений.
- **Система счетов** уже реализована в ADR_MVP как read-only слой поверх данных АБС. Онлайн-кредитование не должно создавать новую read-нагрузку на АБС.
- **Телеком-оператор** используется через существующий СМС-шлюз банка.

#### Логика принятия решений

При принятии решений руководствовался тем, что ADR Task 2/3/4 уже выполнены, и ограничениями из описания задания.

При выборе доработки Системы заявок или АБС выбрал первую, чтобы поддержать границы домена заявок и следованию душителя, чтобы было проще потом перейти на микросервисы. Потенциально после тестирования можно полностью отказаться от работы в АБС и перейти на полностью автоматизированный процесс как с депозитами.

Интернет-портал оставил как есть, как и в ADR_MVP, так как он не распиливался на микросервисы. 

В Кредитном конвейре не менял ядро, так как это коробка, но выбрал доработку обработчика, так как имелась такая возможность.

Для доработки системы кредитного скоринга решил сделать отдельный инстанс, так как это было проще, чем проводить рефакторинг всей системы, выделяя в ней джобы для горизонтального масштабирования.

### <a name="_bjrr7veeh80c"></a>**Альтернативы**

#### Доаотать АБС складывать заявки в кредитный конвейер напрямую

Можно было бы сделать дополнительную интеграцию напрямую, а не через БД, но это только увеличило бы связанность систем.

#### Направлять онлайн-заявки сразу в Кредитный конвейер без передачи заявки в АБС

Такой вариант ускоряет запуск процесса, но создает разрыв с рабочим контуром отделения: менеджер фронт-офиса не сможет продолжить ту же заявку в АБС. Для нового клиента после сайта все равно нужна идентификация в отделении, поэтому заявка должна быть передана в АБС.

#### Рефакторинг системы кредитного скоринга

Переработка системы кредитного скоринга для возможности горизонтального масштабирования сильно увеличила бы скоуп проекта и потребовала бы изменения за пределами кредитной логики.

**Недостатки, ограничения, риски**

- Кредитный конвейер нужно доработать для быстрого запуска онлайн-заявок. Если коробочное BPM-решение или подрядчик ограничат такие изменения, придется использовать менее быстрый путь через АБС, что ухудшит выполнение требования одного дня.
- Синхронизация заявки между системой заявок, АБС и Кредитным конвейером усложняет контроль согласованности. Нужны единый идентификатор, идемпотентные операции и правила обработки дублей.
- Предодобренное предложение не является финальным решением. После подачи заявки скоринг может измениться из-за новых данных, поэтому интерфейс должен явно показывать предварительный характер предложения.
- Для новых клиентов с сайта качество предварительного решения ограничено полнотой данных БКИ и корректностью введенных паспортных данных.
- Изолированный контур предодобрений снижает нагрузку на основной скоринг, но добавляет отдельную эксплуатацию, мониторинг, контроль актуальности данных и правила синхронизации результатов.
- СМС-уведомления зависят от СМС-шлюза и телеком-оператора. При задержках доставки заявка не должна теряться, а клиент должен видеть актуальный статус в цифровом канале.
- Решение не автоматизирует выдачу кредита полностью онлайн для нового клиента: после заявки с сайта клиент должен прийти в отделение для идентификации и получения кредита.
- Кредитное расширение увеличивает ответственность уже созданных в ADR_MVP сервисов заявок, гейтвея и оповещений. Нужно контролировать, чтобы они не превратились в универсальные монолиты цифровой трансформации; при росте сложности потребуется разделение по доменным модулям или сервисам.
